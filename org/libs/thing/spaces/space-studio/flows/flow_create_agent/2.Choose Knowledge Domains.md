Using the agent role definition from the previous step, determine which knowledge domains this agent needs and which fields require user input at runtime.

For each domain:
- Use the domain- prefix followed by the kebab-case domain folder name (e.g., domain-workspace-ops)
- Only attach domains the agent genuinely needs — avoid bloating context with irrelevant knowledge
- Verify that referenced domains exist in the space's knowledge/ directory

For runtime fields:
- Identify fields that MUST be populated by the user before the agent can operate
- Keep runtime fields minimal — only require what the agent cannot function without
- Map each runtime field to its domain and field slug

Consider the agent's pattern when selecting domains:
- Specialists typically need 1-3 focused domains
- Coordinators may need broader domain access to understand what they orchestrate
- Reviewers need domains related to the criteria they validate against
