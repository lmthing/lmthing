# CLAUDE.md — Development Guide for @lmthing/cloud

## Project Overview

**@lmthing/cloud** is the serverless backend for lmthing, providing AI token billing and user management. It runs entirely on **Supabase Edge Functions** (Deno/TypeScript) with **Stripe Token Billing** (`@stripe/ai-sdk`) handling all LLM routing, token metering, and billing.

**Package name:** `@lmthing/cloud`
**Runtime:** Deno (Supabase Edge Functions)
**Database:** Supabase Postgres
**Billing:** Stripe Token Billing (`@stripe/ai-sdk`)
**Monorepo location:** `cloud/` in the `lmthing-monorepo`

## Architecture

```
Frontend → Supabase Edge Function → @stripe/ai-sdk → llm.stripe.com → OpenAI/Anthropic/Google
                 │                        ↓
                 ▼                  Stripe Meter Events
           Supabase Postgres              ↓
          (profiles, api_keys)     Automatic Billing
```

### Core Principle

All LLM requests go through Stripe's proxy at `llm.stripe.com` via `@stripe/ai-sdk`. Stripe handles:
- Provider routing (OpenAI, Anthropic, Google)
- Token counting and metering
- Billing and pricing
- Provider API key management

We only manage: user auth, API keys, and checkout/billing portal flows.

### No Standalone Server

This is NOT a Node.js/Hono server. Each endpoint is an independent Supabase Edge Function running on Deno. There is no `src/` directory, no `build` step, and no `dist/` output.

## Key Files and Their Purpose

### Shared Utilities (`_shared/`)

| File | Exports | Purpose |
|------|---------|---------|
| `auth.ts` | `getUser(req): AuthUser` | Extracts Bearer token from request. If `lmt_` prefix → hash + lookup in `api_keys` table. Otherwise → verify JWT via `supabase.auth.getUser()`. Returns `{ id, email, stripeCustomerId }`. |
| `cors.ts` | `corsHeaders` | Static CORS headers object. Every function must include these and handle `OPTIONS` preflight. |
| `stripe.ts` | `getStripe()`, `ensureStripeCustomer()` | Lazy-initialized Stripe client singleton. `ensureStripeCustomer` creates a Stripe Customer if user doesn't have one yet, saves ID to `profiles`. |
| `supabase.ts` | `createServiceClient()`, `createUserClient(authHeader)` | Service client uses `SUPABASE_SERVICE_ROLE_KEY` (bypasses RLS). User client uses the anon key + user's JWT (respects RLS). |
| `provider.ts` | `resolveModel(modelId, opts?)` | Multi-backend LLM provider abstraction. Selects backend via `LLM_PROVIDER` env var: `"stripe"` (production, routes through llm.stripe.com), `"ollama"` (local/offline), `"openai"` (direct API, no Stripe metering). Default: `"stripe"`. |
| `container.ts` | `getContainer()`, `spaceAppName()`, `SPACE_SPEC`, `HEALTH_CHECK`, `FLY_ORG`, `SPACE_TOKEN_SECRET`, `COMPUTER_TOKEN_SECRET` | Fly.io Machines API abstraction. Provides shared container specs, health checks, token secrets, and app naming conventions for spaces and computers. |

### Edge Functions

| Function | Auth | Method | Key Behavior |
|----------|------|--------|--------------|
| `generate-ai` | JWT or API key | POST | Initializes `createStripe({ apiKey, customerId })` per-request. Supports `stream: true` (returns `toDataStreamResponse()`) and non-streaming (returns OpenAI-compatible JSON). Default model: `openai/gpt-4o-mini`. |
| `list-models` | JWT or API key | GET | Returns hardcoded model list in OpenAI `/v1/models` format. |
| `create-api-key` | JWT only | POST | Generates `lmt_` + 40 hex chars. Hashes with SHA-256 (Web Crypto API). Returns raw key once. |
| `list-api-keys` | JWT only | GET | Returns keys with `prefix`, `name`, `created_at`, `revoked_at` — never the full key. |
| `revoke-api-key` | JWT only | POST | Sets `revoked_at` timestamp (soft delete). Accepts `{ key_id }`. |
| `create-checkout` | JWT | POST | Creates Stripe Checkout session in `subscription` mode. Accepts `{ price_id, success_url, cancel_url }`. |
| `billing-portal` | JWT | POST | Creates Stripe Billing Portal session. Requires user to already have a `stripe_customer_id`. |
| `get-usage` | JWT or API key | GET | Retrieves the Stripe Customer's balance (negative = credit remaining). |
| `stripe-webhook` | None (Stripe signature) | POST | Verifies signature via `constructEventAsync()`. Handles: `billing.alert.triggered`, `checkout.session.completed`, `customer.subscription.*`. Auto-provisions/destroys Fly.io computers on Computer tier subscription changes. |
| `create-sso-code` | JWT only | POST | Generates a single-use SSO authorization code (60s TTL) for cross-domain auth. Accepts `{ redirect_uri, app }`. Returns `{ code }`. |
| `exchange-sso-code` | None (public) | POST | Exchanges SSO code for a Supabase JWT session. Validates code + redirect_uri match. Returns `{ access_token, refresh_token, user }`. |
| `list-spaces` | JWT only | GET | Lists all non-destroyed spaces for the authenticated user, ordered by `created_at` desc. |
| `create-space` | JWT only | POST | Creates a space record and provisions a Fly.io container (app, volume, machine). Accepts `{ name, slug, description?, region? }`. |
| `get-space` | None (public) | GET | Retrieves a single space by slug. Public access for running spaces via RLS. |
| `update-space` | JWT only | PATCH | Updates space metadata (name, description, app_config, auth_enabled, custom_domain). Ownership verified. |
| `start-space` | JWT only | POST | Starts a stopped Fly.io machine for the space. Accepts `{ id }`. |
| `stop-space` | JWT only | POST | Stops a running Fly.io machine for the space. Accepts `{ id }`. |
| `delete-space` | JWT only | POST | Destroys Fly.io resources (machine, volume, app) and sets status to `destroyed`. Accepts `{ id }`. |
| `issue-space-token` | JWT only | POST | Issues a 5-minute HMAC-signed access token for authenticated space WebSocket connections. Accepts `{ spaceId }`. |
| `provision-computer` | JWT only | POST | Provisions a personal Fly.io computer machine (one per user). Creates app, volume, and machine. Accepts `{ region? }`. |
| `issue-computer-token` | JWT only | POST | Issues a 5-minute HMAC-signed access token for computer access. Requires active Computer tier Stripe subscription. |

## Database Schema

Five tables across four migrations — Stripe handles all usage/billing data.

### `profiles`

```sql
create table public.profiles (
  id uuid primary key references auth.users(id) on delete cascade,
  email text not null,
  display_name text,
  stripe_customer_id text unique,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);
```

- Auto-created via `on_auth_user_created` trigger
- `stripe_customer_id` is set lazily on first AI call or checkout
- RLS: users can only SELECT/UPDATE their own row

### `api_keys`

```sql
create table public.api_keys (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references auth.users(id) on delete cascade,
  key_hash text not null unique,
  prefix text not null,
  name text not null default 'Default',
  created_at timestamptz not null default now(),
  revoked_at timestamptz
);
```

- `key_hash`: SHA-256 of the raw `lmt_...` key
- `prefix`: first 12 chars for display
- Partial index on `key_hash WHERE revoked_at IS NULL` for fast lookup
- RLS: users can only access their own keys

### `sso_codes` (002_sso_codes.sql)

```sql
create table public.sso_codes (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references auth.users(id) on delete cascade,
  code text not null unique,
  redirect_uri text not null,
  app text not null,
  expires_at timestamptz not null,
  used_at timestamptz,
  created_at timestamptz not null default now()
);
```

- Single-use authorization codes for cross-domain SSO
- 60-second TTL (`expires_at`)
- No direct user access — only service role can read/write
- Index on `code WHERE used_at IS NULL` for fast lookup

### `spaces` (003_spaces.sql)

```sql
create table public.spaces (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references auth.users(id) on delete cascade,
  slug text not null unique,
  name text not null,
  description text,
  fly_machine_id text unique,
  fly_app_name text,
  fly_volume_id text,
  region text not null default 'iad',
  status text not null default 'created'
    check (status in ('created','provisioning','running','stopped','failed','destroyed')),
  app_config jsonb default '{}',
  auth_enabled boolean default false,
  custom_domain text unique,
  db_schema text,
  internal_key_id uuid references public.api_keys(id),
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  unique (user_id, slug)
);
```

- RLS: owners manage their own spaces; public can read running spaces
- Includes `create_space_schema()` function for per-space schema isolation

### `computers` (004_computers.sql)

```sql
create table public.computers (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null unique references auth.users(id) on delete cascade,
  fly_machine_id text unique,
  fly_app_name text,
  fly_volume_id text,
  region text not null default 'iad',
  status text not null default 'created'
    check (status in ('created','provisioning','running','stopped','failed','destroyed')),
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);
```

- One computer per user (unique `user_id` constraint)
- RLS: users can only manage their own computer

## Authentication Flow

```
Request → Authorization: Bearer <token>
                │
                ├── starts with "lmt_"?
                │   ├── YES → SHA-256 hash → lookup in api_keys → get user_id → get profile
                │   └── NO  → supabase.auth.getUser(token) → get profile
                │
                └── Returns: { id, email, stripeCustomerId }
```

Both paths resolve to the same `AuthUser` interface, so all downstream code is auth-method-agnostic.

## Billing Flow (How @stripe/ai-sdk Works)

```typescript
// In generate-ai/index.ts:
const stripeLLM = createStripe({
  apiKey: Deno.env.get("STRIPE_SECRET_KEY")!,  // Your Stripe key — NOT an LLM provider key
  customerId: stripeCustomerId,                  // Bill this Stripe Customer
});

const result = await streamText({
  model: stripeLLM("openai/gpt-4o-mini"),       // Routes through llm.stripe.com
  messages,
});
```

1. `createStripe()` creates a Vercel AI SDK provider backed by Stripe's LLM proxy
2. `stripeLLM("openai/gpt-4o-mini")` returns a model that sends requests to `llm.stripe.com`
3. Stripe forwards to the actual provider (OpenAI in this case)
4. Stripe automatically records meter events for input/output tokens
5. Billing happens according to your Stripe Token Billing configuration

### What Stripe Sends (per request)

```json
{ "event_name": "token-billing-tokens", "payload": { "stripe_customer_id": "cus_xxx", "value": "150", "model": "openai/gpt-4o-mini", "token_type": "input" } }
{ "event_name": "token-billing-tokens", "payload": { "stripe_customer_id": "cus_xxx", "value": "42", "model": "openai/gpt-4o-mini", "token_type": "output" } }
```

### Model Naming Convention

Models use `provider/model-name` format through Stripe's proxy:
- `openai/gpt-4o`, `openai/gpt-4o-mini`, `openai/gpt-4.1`, `openai/o3`
- `anthropic/claude-opus-4`, `anthropic/claude-sonnet-4`, `anthropic/claude-3-5-haiku`
- `google/gemini-2.5-pro`, `google/gemini-2.5-flash`

### Provider Limitation

`@stripe/ai-sdk` provider does NOT support tool calling. If tool use is needed in the future, use the `meteredModel` approach instead, which wraps any AI SDK provider.

## Coding Conventions

### Edge Function Pattern

Every function follows this structure:

```typescript
import { corsHeaders } from "../_shared/cors.ts";
import { getUser } from "../_shared/auth.ts";

Deno.serve(async (req) => {
  // 1. CORS preflight
  if (req.method === "OPTIONS") {
    return new Response("ok", { headers: corsHeaders });
  }

  try {
    // 2. Auth (skip for stripe-webhook)
    const user = await getUser(req);

    // 3. Business logic
    // ...

    // 4. Success response
    return new Response(JSON.stringify({ ... }), {
      headers: { ...corsHeaders, "Content-Type": "application/json" },
    });
  } catch (err) {
    // 5. Error response
    const message = err instanceof Error ? err.message : "Internal server error";
    return new Response(JSON.stringify({ error: { message } }), {
      status: 500,
      headers: { ...corsHeaders, "Content-Type": "application/json" },
    });
  }
});
```

### Import Conventions (Deno)

```typescript
// npm packages use npm: specifier
import { createStripe } from "npm:@stripe/ai-sdk@0.1/provider";
import { streamText } from "npm:ai@5";
import Stripe from "npm:stripe@17";
import { createClient } from "npm:@supabase/supabase-js@2";

// Shared modules use relative .ts imports
import { corsHeaders } from "../_shared/cors.ts";
import { getUser } from "../_shared/auth.ts";
```

### Error Handling

- Auth errors: return 401
- Validation errors: return 400
- Not found: return 404
- All others: return 500
- Error format: `{ error: { message: string } }`
- Stripe webhook: return 400 for invalid signature, 200 for all handled/unhandled events

### API Key Format

- Prefix: `lmt_`
- Key body: 40 hex characters (20 random bytes)
- Full key: `lmt_` + 40 hex = 44 characters total
- Display prefix: first 12 characters
- Storage: SHA-256 hash via Web Crypto API (`crypto.subtle.digest`)
- Revocation: soft delete via `revoked_at` timestamp

### Stripe Customer Lifecycle

1. User signs up → profile created (no `stripe_customer_id` yet)
2. First `generate-ai` or `create-checkout` call → `ensureStripeCustomer()` creates Stripe Customer and saves ID
3. All subsequent calls use the stored `stripe_customer_id`

## Adding a New Edge Function

1. Create directory: `supabase/functions/<function-name>/`
2. Create `index.ts` following the standard pattern above
3. Import from `../_shared/` for auth, CORS, Stripe, Supabase
4. Deploy: `pnpx supabase functions deploy <function-name>`
5. If the function should NOT require JWT (like webhooks): deploy with `--no-verify-jwt`
6. Update `README.md` API Reference section

## Adding a New Database Table

1. Create `supabase/migrations/<NNN>_<description>.sql`
2. Define the table with appropriate columns and types
3. Enable RLS: `alter table public.<table> enable row level security;`
4. Add RLS policies for user access
5. Add indexes for common query patterns
6. Push: `pnpx supabase db push`
7. Update the Database Schema section in `README.md`

## Important Implementation Details

### config.toml Format

The Supabase config uses `project_id = "..."` as a top-level field, NOT a `[project]` section. The file is generated by `supabase init` — don't manually create it with the wrong format.

### Webhook Security

The `stripe-webhook` function is the only function without user auth. It verifies requests using Stripe's webhook signature (`constructEventAsync`). It must be deployed with `--no-verify-jwt`.

### Streaming vs Non-streaming

`generate-ai` supports both modes:
- `stream: false` (default): Returns OpenAI-compatible JSON with `choices[0].message.content` and `usage`
- `stream: true`: Returns Vercel AI SDK data stream via `result.toDataStreamResponse()`

### Edge Function Timeout

Supabase Edge Functions timeout after 60 seconds. For long LLM generations, always use `stream: true` — streaming starts returning data immediately and keeps the connection alive.

### No Usage Tables

Unlike traditional architectures, we do NOT maintain a `usage_logs` table. All token metering is handled by Stripe's meter events automatically. The `get-usage` function queries the Stripe Customer balance directly.
