# `project.json` — project-app descriptor

A small JSON file at a project-app's root carrying browse/identity metadata for the store catalog `store/scripts/gen-apps-manifest.mjs:141-166`. Metadata for a catalog entry comes from `project.json` when present, otherwise it is derived from the app's [`package.json`](./package.json.md) — **either is accepted** `store/scripts/gen-apps-manifest.mjs:147-148`. Only the hand-authored `demo-feed` catalog app ships a standalone `project.json`; the other catalog apps (`blog`, `health`, `kitchen`, `trips`, `homes`) carry their identity on `package.json` instead `store/projects/demo-feed/project.json` and `store/projects/blog/package.json`.

## Format

```json
{
  "id": "demo-feed",
  "title": "Personal Feed",
  "description": "Hand-authored reference app for the project-as-application engine: a personal reading feed with a database table, list/mark-read/add-item API endpoints, a page, and a database hook that enriches new items.",
  "icon": "📰"
}
```

(Verbatim from the real example file `store/projects/demo-feed/project.json`.)

## Fields

| Field | Type | Notes |
|---|---|---|
| `id` | string | The app id; falls back to the on-disk directory name when absent `store/scripts/gen-apps-manifest.mjs:163`. It is also the install/materialization id — the pod installs a catalog app into `<lmthingRoot>/<projectId ?? appId>/` `sdk/org/libs/cli/src/server/routes/apps.ts:176-177,210`. |
| `title` | string | Human display name; falls back to a title-cased `id` when absent `store/scripts/gen-apps-manifest.mjs:164`. |
| `description` | string | One-line catalog summary; falls back to `package.json`'s `description` then `''` `store/scripts/gen-apps-manifest.mjs:165`. |
| `icon` | string \| null | Emoji shown on the catalog card; `null` when no `project.json` (there is no `package.json` fallback for `icon`) `store/scripts/gen-apps-manifest.mjs:166`. |

## Consumed by the catalog manifest

These four fields are surfaced per-app into the generated browse index `store/projects/manifest.json` (`{apps:[{id,title,description,icon,tables,pages,endpoints,hooks,files}]}`) `store/scripts/gen-apps-manifest.mjs:162-174`. The manifest is generated by scanning `store/projects/<id>/` and must never be hand-edited — it is regenerated by the `lmthing-apps-manifest` Vite plugin on build `store/vite.config.ts` `appsManifestPlugin`. The manifest is read at runtime through the typed accessor `listCatalogApps()`/`getCatalogApp(id)` over the inlined data `store/src/lib/apps-manifest.ts` `CatalogApp`.

## Synthesized on install when absent

Because app-builder-authored apps ship a `package.json` but no `project.json`, the pod's install route synthesizes a **deterministic** `project.json` (`id`/`title`/`description`/`icon` from the catalog entry, no volatile fields) into the installed project directory when one is absent `sdk/org/libs/cli/src/server/routes/apps.ts:155-165`. Determinism matters because `project.json` is part of the app-template hash, so a volatile field would make every re-install look diverged `sdk/org/libs/cli/src/server/routes/apps.ts:149-153`.

## The `id` in the served URL

The same id names the app's URL. The pod mounts every installed project's pages at `/app/<projectId>/*` and its api at `/app/<projectId>/api/*` `sdk/org/libs/cli/src/server/serve.ts:218,306`. On production `lmthing.app` the same pod *additionally* root-mounts the app at a bare `/<projectId>/*` for clean URLs — gated on `LMTHING_GATEWAY_URL`, which is present exactly when the pod sits behind the Envoy shell/pod split, because a bare `/:projectId/*` would otherwise shadow every route of the unified SPA that this server also serves in local single-serve `sdk/org/libs/cli/src/server/serve.ts:308-326`. The web app mirrors the split with `APP_PATH_PREFIX` — `''` on hostname `lmthing.app`, `'/app'` everywhere else `sdk/org/apps/web/src/lib/config.ts:28-39` — and the launcher opens `<COMPUTER_BASE_URL><APP_PATH_PREFIX>/<projectId>/` `sdk/org/apps/web/src/routes/apps/index.tsx:19-22`.
