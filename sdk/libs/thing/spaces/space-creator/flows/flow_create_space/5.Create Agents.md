Design and generate 2-3 agents for this space. Each agent must have a distinct, non-overlapping role that covers one or more of the space's core use cases identified in the space brief.

For each agent, generate three files:

**instruct.md** — The agent's identity and instructions:
- Frontmatter: `name` (PascalCase), `description` (one sentence), `tools` (array), `enabledKnowledgeFields` (array with `domain-` prefix matching knowledge folder names)
- Slash actions linking to flows with correct `flowId` references
- System prompt using the appropriate strategy: Identity-First (default for specialists), Constraint-Driven (for reviewers/validators), or Knowledge-Referencing (for agents with many domains)
- Clear boundaries on what the agent does and does NOT do

**config.json** — Runtime field requirements:
- `runtimeFields` mapping domain names to arrays of field names that should prompt the user at conversation start

**values.json** — Always starts as `{}`

Design principles:
- Agents should complement each other with distinct roles (e.g., one plans, one executes, one reviews)
- Each agent should have 2-4 enabled knowledge domains relevant to its role
- Slash actions should reference flows that will be created in the next step
- Runtime fields should be limited to fields the agent truly needs before starting

Validate all cross-references: `enabledKnowledgeFields` values must match knowledge folder names with `domain-` prefix, `flowId` values must follow `flow_` + snake_case convention.
