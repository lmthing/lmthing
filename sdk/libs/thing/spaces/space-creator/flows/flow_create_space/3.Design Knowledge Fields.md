For each domain from the previous step, design 1-3 fields that represent distinct dimensions of that domain's knowledge.

For each field, determine:
- **Folder name** — kebab-case (e.g., `experience-level`)
- **Label** — Human-readable (e.g., "Experience Level")
- **Field type** — One of: `select` (one choice from a list), `multiSelect` (multiple choices), `text` (free-form string), `number` (numeric value)
- **Variable name** — camelCase conversion of the folder name (e.g., `experienceLevel`). Must be unique across all domains.
- **Required** — Set to `true` only if the agent cannot function without this value. Use sparingly.
- **Default value** — The most common or safest choice. For select fields, must match an option slug.
- **Options** — For select/multiSelect fields, list 3-5 option names with kebab-case slugs.

Follow these design principles:
- One field per decision the agent needs to make within this domain
- Fields should be orthogonal — selecting one field's value should not determine another's
- Prefer `select` over `text` when possible, as select fields provide rich option content that shapes agent behavior
- If a domain only needs one field, that is fine — do not add fields for completeness

Validate all names against naming conventions: field folders in kebab-case, variable names in camelCase, option slugs in kebab-case.
